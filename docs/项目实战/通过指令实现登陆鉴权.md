# 从业务耦合到可复用：实现一个 Vue3 登录鉴权指令

在前端项目里，“未登录拦截”几乎是一个绕不开的需求。点赞、收藏、评论、下单，甚至某些页面操作，往往都要求用户先登录。

很多项目在最初实现时，会把这段逻辑直接写进组件里，或者封装成一个和当前业务强绑定的指令。这样的方式短期能跑起来，但后面通常会遇到两个问题：一是难以复用，二是越来越难维护。

这篇文章想解决的就是这个问题：把一个和业务深度绑定的登录拦截逻辑，改造成一个可以在 Vue3 项目里复用的通用指令。

## 原始实现有什么问题

常见的写法大概像这样：

```ts
const authStore = useAuthStore();

el.addEventListener("click", (event) => {
  if (!authStore.isLogin) {
    authStore.userInfos.loginVisible = true;
    event.stopImmediatePropagation();
    event.preventDefault();
  }
});
```

这段代码能工作，但它把几个本不该耦合在一起的职责绑死了。

- 指令直接依赖 `store`，换一个项目就很难复用。
- 指令直接控制 UI，例如打开登录弹窗，这其实已经越过了“指令”的职责边界。
- 后续扩展困难，比如“登录成功后继续刚才的操作”，实现起来会越来越别扭。

问题的根源在于：指令既做了“拦截”，又做了“业务决策”。

## 设计目标

更合理的设计应该是：

> 指令只负责判断和拦截，不负责具体业务。

也就是说，这个指令本身不关心：

- 你用的是 Pinia、Vuex，还是本地缓存。
- 登录失败时是弹窗、抽屉，还是跳转登录页。
- 登录成功后要继续执行什么操作。

这些都应该由外部传入，指令只做一件事：在合适的时机拦截事件。

## 核心思路：把业务能力外置

原本写死在指令里的逻辑可能是这样：

```ts
authStore.isLogin;
authStore.userInfos.loginVisible = true;
```

改造之后，指令只依赖传入参数：

```ts
binding.value.isLogin;
binding.value.onAuthFail;
```

这个变化看起来很小，但职责边界已经完全不同了。

- `isLogin` 用来判断当前是否已登录。
- `onAuthFail` 用来定义“未登录时应该怎么处理”。
- `onAuthSuccess` 用来定义“已登录时是否要做额外处理”。
- `afterLogin` 用来实现“登录成功后继续操作”。

这样一来，指令就从“业务逻辑的一部分”变成了“通用能力的载体”。

## Vue3 通用指令实现

下面是一版可以直接在 Vue3 中使用的实现：

```ts
// directives/auth.ts
import type { DirectiveBinding, ObjectDirective } from "vue";

const defaultEvents = [
  "click",
  "mousedown",
  "mouseup",
  "keydown",
  "keyup",
  "focus",
];

type AuthBinding = {
  isLogin: boolean | (() => boolean);
  onAuthFail?: (event: Event) => void;
  onAuthSuccess?: (event: Event) => void;
  afterLogin?: () => void;
  events?: string[];
};

type AuthElement = HTMLElement & {
  __authHandler__?: EventListener;
};

export const authDirective: ObjectDirective<AuthElement, AuthBinding> = {
  mounted(el, binding: DirectiveBinding<AuthBinding>) {
    const {
      isLogin,
      onAuthFail,
      onAuthSuccess,
      afterLogin,
      events = defaultEvents,
    } = binding.value || {};

    const checkLogin = () =>
      typeof isLogin === "function" ? isLogin() : Boolean(isLogin);

    const handler = (event: Event) => {
      if (!checkLogin()) {
        event.stopImmediatePropagation();
        event.preventDefault();

        onAuthFail?.(event);

        if (typeof afterLogin === "function") {
          const loginHandler = () => {
            afterLogin();
            document.removeEventListener("auth:login", loginHandler);
          };

          document.addEventListener("auth:login", loginHandler);
        }

        return;
      }

      onAuthSuccess?.(event);
    };

    el.__authHandler__ = handler;

    events.forEach((eventType) => {
      el.addEventListener(eventType, handler, { capture: true });
    });
  },

  unmounted(el, binding) {
    const events = binding.value?.events || defaultEvents;

    events.forEach((eventType) => {
      if (el.__authHandler__) {
        el.removeEventListener(eventType, el.__authHandler__, {
          capture: true,
        });
      }
    });

    delete el.__authHandler__;
  },
};
```

这版实现里，指令本身不依赖任何业务模块，只接受外部传入的判断条件和回调函数，因此可以在不同项目里重复使用。

## 注册方式

在 Vue3 入口文件中注册即可：

```ts
// main.ts
import { createApp } from "vue";
import App from "./App.vue";
import { authDirective } from "./directives/auth";

const app = createApp(App);

app.directive("auth", authDirective);

app.mount("#app");
```

## 使用方式

### 基础用法

最基础的场景就是：未登录时打开登录弹窗。

```vue
<button
  v-auth="{
    isLogin: () => authStore.isLogin,
    onAuthFail: () => openLogin(),
  }"
>
  点我
</button>
```

这里的 `openLogin` 是业务侧函数，指令本身并不关心它的实现。

### 登录后继续执行操作

如果希望用户登录完成后自动继续原来的动作，可以这样写：

```vue
<button
  v-auth="{
    isLogin: () => authStore.isLogin,
    onAuthFail: openLogin,
    afterLogin: () => submit(),
  }"
>
  提交
</button>
```

登录成功后，只需要在合适的时机派发一个事件：

```ts
document.dispatchEvent(new Event("auth:login"));
```

这样就能把“登录成功后继续执行”的能力从具体业务流程里拆出来。

### 自定义拦截事件

如果你只想拦截点击，而不处理其他事件，也可以把事件列表显式传入：

```vue
<div
  v-auth="{
    isLogin,
    onAuthFail,
    events: ['click'],
  }"
/>
```

## 几个关键设计点

### 1. 为什么使用捕获阶段

注册监听时使用了：

```ts
{ capture: true }
```

这样可以在事件进入目标元素之前就完成拦截，避免组件内部已经先执行了自己的点击逻辑。对于“未登录时禁止继续触发后续行为”这种场景，捕获阶段通常更稳妥。

### 2. 为什么使用 `stopImmediatePropagation`

这里没有只用 `stopPropagation`，而是用了更强的：

```ts
event.stopImmediatePropagation();
```

原因是我们不仅希望事件不要继续向外传播，还希望当前元素上其他同类监听器也不要再执行。这样才能真正做到“拦截之后不再继续业务流程”。

### 3. 为什么不用事件总线

示例里用的是：

```ts
document.dispatchEvent(new Event("auth:login"));
```

这么做的优点是：

- 不依赖第三方事件库。
- 可以跨组件通信。
- 在简单场景里足够轻量。

如果你的项目本身已经有统一的事件系统，当然也可以替换成自己的实现。重点不在于一定要用 `document`，而在于把“登录成功后的触发机制”从指令内部解耦出来。

## 可以继续扩展成权限指令

这个指令的设计不只适用于“是否登录”，它也很适合继续扩展成“登录 + 权限”统一指令。

例如：

```vue
<button
  v-auth="{
    isLogin,
    permission: 'user.delete',
    onNoPermission: () => toast('无权限'),
  }"
>
  删除
</button>
```

对应的内部判断也可以继续扩展：

```ts
if (!isLogin) {
  // 未登录
} else if (!hasPermission) {
  // 无权限
}
```

这样一来，登录拦截和权限控制就能复用同一套指令设计思路。

## 总结

这个方案的核心不在于“写了一个指令”，而在于明确了指令的职责边界：

- 指令负责判断和拦截。
- 登录弹窗、页面跳转、埋点、继续执行等行为全部交给业务层。

当职责被拆开之后，代码的可复用性、可维护性和可扩展性都会明显提升。对于 Vue3 项目来说，这种“能力外置”的写法，通常也比把业务直接揉进指令里更健康。
