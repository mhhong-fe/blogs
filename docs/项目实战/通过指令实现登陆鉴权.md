# 从业务耦合到可复用：实现一个 Vue3 登录鉴权指令

在前端项目里，“未登录拦截”几乎是一个绕不开的需求。点赞、收藏、评论、下单，甚至某些页面操作，往往都要求用户先登录。

很多项目在最初实现时，会把这段逻辑直接写进组件里，或者封装成一个和当前业务强绑定的指令。这样的方式短期能跑起来，但后面通常会遇到两个问题：一是难以复用，二是越来越难维护。

这篇文章想解决的就是这个问题：把一个和业务深度绑定的登录拦截逻辑，改造成一个可以在 Vue3 项目里复用的通用指令。

## 原始实现有什么问题

常见的写法大概像这样：

```ts
const authStore = useAuthStore();

el.addEventListener("click", (event) => {
  if (!authStore.isLogin) {
    authStore.userInfos.loginVisible = true;
    event.stopImmediatePropagation();
    event.preventDefault();
  }
});
```

这段代码能工作，但它把几个本不该耦合在一起的职责绑死了。

- 指令直接依赖 `store`，换一个项目就很难复用。
- 指令直接控制 UI，例如打开登录弹窗，这其实已经越过了“指令”的职责边界。
- 后续扩展困难，比如“登录成功后继续刚才的操作”，实现起来会越来越别扭。

问题的根源在于：指令既做了“拦截”，又做了“业务决策”。

## 设计目标

更合理的设计应该是：

> 指令只负责判断和拦截，不负责具体业务。

但这个目标其实可以分成两个阶段来实现：

1. 先做一个完全业务无关的通用指令。
2. 再解决“通用是通用了，但每个组件都要传一大坨参数”的问题。

这样设计的好处是，抽象过程更清晰。我们先把耦合拆开，再讨论怎样把重复配置收拢回项目层。

## 第一阶段：先实现一个业务无关的通用指令

第一步不要急着谈全局配置，而是先把指令的职责压缩到最小。

原本写死在指令里的逻辑可能是这样：

```ts
authStore.isLogin;
authStore.userInfos.loginVisible = true;
```

如果把它改造成真正通用的指令，应该只保留“判断 + 拦截 + 回调”这几个能力：

```ts
binding.value.isLogin;
binding.value.onAuthFail;
binding.value.onAuthSuccess;
binding.value.afterLogin;
```

也就是说：

- `isLogin` 负责告诉指令当前是否已登录。
- `onAuthFail` 负责定义未登录时该做什么。
- `onAuthSuccess` 负责定义已登录时的补充行为。
- `afterLogin` 用来实现“登录成功后继续刚才的操作”。

这样一来，指令本身不依赖任何 store，也不控制任何 UI，真正做到了业务无关。

## 纯通用版实现

下面先看第一版，也就是完全业务无关的实现。

```ts
// directives/auth.ts
import type { DirectiveBinding, ObjectDirective } from "vue";

const defaultEvents = [
  "click",
  "mousedown",
  "mouseup",
  "keydown",
  "keyup",
  "focus",
];

type AuthBinding = {
  isLogin: boolean | (() => boolean);
  onAuthFail?: (event: Event) => void;
  onAuthSuccess?: (event: Event) => void;
  afterLogin?: () => void;
  events?: string[];
};

type AuthElement = HTMLElement & {
  __authHandler__?: EventListener;
};

export const authDirective: ObjectDirective<AuthElement, AuthBinding> = {
  mounted(el, binding: DirectiveBinding<AuthBinding>) {
    const {
      isLogin,
      onAuthFail,
      onAuthSuccess,
      afterLogin,
      events = defaultEvents,
    } = binding.value || {};

    const checkLogin = () =>
      typeof isLogin === "function" ? isLogin() : Boolean(isLogin);

    const handler = (event: Event) => {
      if (!checkLogin()) {
        event.stopImmediatePropagation();
        event.preventDefault();

        onAuthFail?.(event);

        if (typeof afterLogin === "function") {
          const loginHandler = () => {
            afterLogin();
            document.removeEventListener("auth:login", loginHandler);
          };

          document.addEventListener("auth:login", loginHandler);
        }

        return;
      }

      onAuthSuccess?.(event);
    };

    el.__authHandler__ = handler;

    events.forEach((eventType) => {
      el.addEventListener(eventType, handler, { capture: true });
    });
  },

  unmounted(el, binding) {
    const events = binding.value?.events || defaultEvents;

    events.forEach((eventType) => {
      if (el.__authHandler__) {
        el.removeEventListener(eventType, el.__authHandler__, {
          capture: true,
        });
      }
    });

    delete el.__authHandler__;
  },
};
```

在 Vue3 里注册也很直接：

```ts
// main.ts
import { createApp } from "vue";
import App from "./App.vue";
import { authDirective } from "./directives/auth";

const app = createApp(App);

app.directive("auth", authDirective);

app.mount("#app");
```

## 纯通用版怎么使用

### 基础用法

```vue
<button
  v-auth="{
    isLogin: () => authStore.isLogin,
    onAuthFail: () => openLoginModal(),
  }"
>
  点赞
</button>
```

### 登录后继续执行操作

```vue
<button
  v-auth="{
    isLogin: () => authStore.isLogin,
    onAuthFail: () => openLoginModal(),
    afterLogin: () => submit(),
  }"
>
  提交订单
</button>
```

登录成功后，只需要在合适的时机派发一个事件：

```ts
document.dispatchEvent(new Event("auth:login"));
```

### 自定义拦截事件

```vue
<div
  v-auth="{
    isLogin: () => authStore.isLogin,
    onAuthFail: () => openLoginModal(),
    events: ['click'],
  }"
/>
```

## 通用版的问题：复用是有了，但组件传参太重

走到这一步，指令已经足够通用了，但新的问题也很明显：每个组件都要重复传很多相似参数。

比如在一个真实项目里，下面这些内容往往是全站一致的：

- 登录态判断都来自 `authStore.isLogin`
- 未登录时都要上报埋点
- 上报完成后都要弹出登录弹窗

如果每个按钮都这么写：

```vue
<button
  v-auth="{
    isLogin: () => authStore.isLogin,
    onAuthFail: () => openLoginModal(),
  }"
>
  点赞
</button>
```

那虽然它是“通用”的，但使用成本并不低。

这时候就可以进入第二阶段：保留通用指令的能力边界，再把项目级公共逻辑通过全局配置收拢起来。

## 第二阶段：在通用指令之上引入全局配置

这个阶段不是推翻第一版，而是在第一版的基础上做增强。

我们把这些项目级通用能力提到安装阶段统一注入：

- 默认 `isLogin`
- 默认 `onAuthFail`
- 默认 `onAuthSuccess`

同时，再加入一个 `scene` 参数，用来标记这次鉴权发生在哪个业务场景，例如：

- `like`
- `comment`
- `submit-order`
- `collect`

这样默认失败处理里就可以统一根据 `scene` 上报埋点，再弹出登录弹窗。

## 全局配置版实现

```ts
// directives/auth.ts
import type { App, DirectiveBinding, ObjectDirective } from "vue";

const defaultEvents = [
  "click",
  "mousedown",
  "mouseup",
  "keydown",
  "keyup",
  "focus",
];

type AuthContext = {
  event: Event;
  scene?: string;
  el: HTMLElement;
};

type AuthBinding = {
  isLogin?: boolean | (() => boolean);
  scene?: string;
  onAuthFail?: (context: AuthContext) => void;
  onAuthSuccess?: (context: AuthContext) => void;
  afterLogin?: () => void;
  events?: string[];
};

type AuthDirectiveOptions = {
  isLogin?: boolean | (() => boolean);
  onAuthFail?: (context: AuthContext) => void;
  onAuthSuccess?: (context: AuthContext) => void;
};

type AuthElement = HTMLElement & {
  __authHandler__?: EventListener;
};

export function createAuthDirective(options: AuthDirectiveOptions = {}) {
  const authDirective: ObjectDirective<AuthElement, AuthBinding> = {
    mounted(el, binding: DirectiveBinding<AuthBinding>) {
      const {
        isLogin,
        scene,
        onAuthFail,
        onAuthSuccess,
        afterLogin,
        events = defaultEvents,
      } = binding.value || {};

      const loginChecker = isLogin ?? options.isLogin;

      const checkLogin = () =>
        typeof loginChecker === "function"
          ? loginChecker()
          : Boolean(loginChecker);

      const handler = (event: Event) => {
        const context = {
          event,
          scene,
          el,
        };

        if (!checkLogin()) {
          event.stopImmediatePropagation();
          event.preventDefault();

          const failHandler = onAuthFail ?? options.onAuthFail;
          failHandler?.(context);

          if (typeof afterLogin === "function") {
            const loginHandler = () => {
              afterLogin();
              document.removeEventListener("auth:login", loginHandler);
            };

            document.addEventListener("auth:login", loginHandler);
          }

          return;
        }

        const successHandler = onAuthSuccess ?? options.onAuthSuccess;
        successHandler?.(context);
      };

      el.__authHandler__ = handler;

      events.forEach((eventType) => {
        el.addEventListener(eventType, handler, { capture: true });
      });
    },

    unmounted(el, binding) {
      const events = binding.value?.events || defaultEvents;

      events.forEach((eventType) => {
        if (el.__authHandler__) {
          el.removeEventListener(eventType, el.__authHandler__, {
            capture: true,
          });
        }
      });

      delete el.__authHandler__;
    },
  };

  return authDirective;
}

export function installAuthDirective(
  app: App,
  options: AuthDirectiveOptions = {},
) {
  app.directive("auth", createAuthDirective(options));
}
```

这版实现的关键在于优先级：

- 先用组件局部传入的配置
- 局部没传时，回退到全局默认配置

这样既保留了通用性，也解决了重复传参的问题。

## 全局配置版注册方式

```ts
// main.ts
import { createApp } from "vue";
import App from "./App.vue";
import { installAuthDirective } from "./directives/auth";

const app = createApp(App);

installAuthDirective(app, {
  isLogin: () => authStore.isLogin,
  onAuthFail({ scene }) {
    tracking("auth_fail", {
      scene,
    });

    openLoginModal();
  },
  onAuthSuccess({ scene }) {
    tracking("auth_success", {
      scene,
    });
  },
});

app.mount("#app");
```

这里的 `tracking` 只作为埋点函数示意，不需要在指令里实现。`openLoginModal` 也由业务项目自己提供。

## 全局配置版怎么使用

### 基础用法

有了全局默认配置之后，大多数场景只需要传 `scene`：

```vue
<button
  v-auth="{
    scene: 'like',
  }"
>
  点赞
</button>
```

当用户未登录时，默认失败钩子就会拿到 `scene: 'like'`，统一做埋点和弹窗。

### 局部覆盖默认失败处理

```vue
<button
  v-auth="{
    scene: 'comment',
    onAuthFail: ({ scene }) => {
      tracking('comment_auth_fail', { scene });
      openCommentLoginModal();
    },
  }"
>
  评论
</button>
```

### 登录后继续执行操作

```vue
<button
  v-auth="{
    scene: 'submit-order',
    afterLogin: () => submit(),
  }"
>
  提交订单
</button>
```

### 自定义拦截事件

```vue
<div
  v-auth="{
    scene: 'collect',
    events: ['click'],
  }"
/>
```

### 局部覆盖默认登录态判断

大多数场景会直接使用全局 `isLogin`，但如果某个页面的登录态判断来源不同，也可以单独覆盖：

```vue
<button
  v-auth="{
    isLogin: () => subAppAuthStore.isLogin,
    scene: 'special-action',
  }"
>
  特殊操作
</button>
```

## 几个关键设计点

### 1. 为什么拆分指令中的业务逻辑
* 避免指令拦截功能和业务逻辑耦合，更加通用
* 通过installAuthDirective对指令封装一层，全局传入默认登陆判断、默认失败处理，又维持了业务组件使用指令的干净、便捷

### 2. 为什么使用捕获阶段

注册监听时使用了：

```ts
{ capture: true }
```

这样可以在事件进入目标元素之前就完成拦截，避免组件内部已经先执行了自己的点击逻辑。对于“未登录时禁止继续触发后续行为”这种场景，捕获阶段通常更稳妥。

### 3. 为什么使用 `stopImmediatePropagation`

这里没有只用 `stopPropagation`，而是用了更强的：

```ts
event.stopImmediatePropagation();
```

原因是我们不仅希望事件不要继续向外传播，还希望当前元素上其他同类监听器也不要再执行。这样才能真正做到“拦截之后不再继续业务流程”。

### 4. 为什么不用事件总线

示例里用的是：

```ts
document.dispatchEvent(new Event("auth:login"));
```

这么做的优点是：

- 不依赖第三方事件库。
- 可以跨组件通信。
- 在简单场景里足够轻量。

如果你的项目本身已经有统一的事件系统，当然也可以替换成自己的实现。重点不在于一定要用 `document`，而在于把“登录成功后的触发机制”从指令内部解耦出来。

## 可以继续扩展成权限指令

这套设计不只适用于“是否登录”，它也很适合继续扩展成“登录 + 权限”统一指令。

例如：

```vue
<button
  v-auth="{
    isLogin,
    scene: 'delete-user',
    permission: 'user.delete',
    onNoPermission: () => toast('无权限'),
  }"
>
  删除
</button>
```

对应的内部判断也可以继续扩展：

```ts
if (!isLogin) {
  // 未登录
} else if (!hasPermission) {
  // 无权限
}
```

这样一来，登录拦截、场景埋点和权限控制就能复用同一套指令设计思路。

## 总结

这套设计其实是在回答两个问题：

第一，怎样做一个业务无关的登录鉴权指令。答案是先把能力压缩成 `isLogin`、`onAuthFail`、`onAuthSuccess`、`afterLogin` 这些最小能力。

第二，怎样让这个通用指令在真实项目里用起来不啰嗦。答案是在不破坏职责边界的前提下，引入全局默认配置，把默认 `isLogin`、默认失败处理、默认成功处理统一收拢到项目层。

这样一来，指令仍然是通用的，但组件层的使用成本会低很多。大多数场景只需要传一个 `scene`，真正特殊的场景再局部覆盖即可。
