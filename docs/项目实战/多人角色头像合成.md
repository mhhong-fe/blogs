# 用 Vue + Canvas 实现一个多人角色头像合成器

> 本文记录一个「多人角色头像合成」案例的完整实现思路：用户选择 2 到 4 个角色头像后，可以切换拼图模板，并在 Canvas 内拖拽、缩放头像，最后导出一张高清合成图。

这个需求乍看只是“把几张图片拼到一起”，但真正做起来会遇到不少细节：不规则裁剪区域怎么配置？图片如何保持覆盖裁剪区？PC 和移动端怎么共用拖拽逻辑？预览画布和导出清晰度如何分离？本文会基于一个 Nuxt/Vue 实现，拆解其中的关键设计。

## 最终效果

功能上主要包括：

- 选择 2 到 4 个头像参与合成。
- 根据选择人数自动匹配可用模板。
- 每个模板由多个多边形裁剪区域组成。
- Canvas 中每张头像可以单独拖拽调整位置。
- PC 支持滚轮缩放，移动端支持双指捏合缩放。
- 拖拽到其他区域时可以交换头像顺序。
- 导出时使用离屏 Canvas 输出高清 PNG。

整体结构可以拆成三层：

```text
index.vue
  ├─ 负责页面状态、头像选择、模板切换
  ├─ 调用 useMultiAvatarCanvas
  └─ 渲染 canvas 预览

useMultiAvatarCanvas.ts
  ├─ 负责 Canvas 绘制
  ├─ 负责拖拽、缩放、交换手势
  ├─ 负责图片缓存和高清导出
  └─ 对外暴露 render / reset / download 等能力

config.ts
  ├─ 头像资源
  ├─ 模板缩略图
  └─ 多边形裁剪路径配置
```

## 技术选型：为什么用原生 Canvas 逻辑实现

做 Canvas 交互类需求时，很自然会想到 Fabric.js、Konva、PixiJS 这类第三方库。它们都很成熟，尤其在对象选择、图层管理、变换控制、复杂编辑器场景里能节省大量工作。

但这个头像合成器最后选择了「Vue 状态管理 + 原生 Canvas API」的方式，主要原因是：这个需求的核心不是通用画布编辑器，而是一个边界清晰、规则固定的图片合成工具。

具体来说，它有几个特点：

- 画布尺寸固定，逻辑坐标只有 `150 x 150`。
- 可编辑对象数量很少，最多 4 张头像。
- 图层结构稳定：每个头像只属于一个模板槽位。
- 交互能力明确：拖拽、缩放、交换、导出。
- 裁剪区域由模板多边形决定，不需要用户自由绘制路径。
- 导出结果要求干净，不需要保留编辑器控制框、选中框等通用编辑状态。

如果引入 Fabric 或 Konva，当然也能实现，但会带来一些额外复杂度：

1. 需要把模板多边形转换成库里的对象或 clip 配置。
2. 需要处理库对象坐标系和我们模板逻辑坐标系之间的映射。
3. 需要定制对象控制点，避免出现不符合业务预期的旋转、拉伸、选择框。
4. 导出时仍然要确保裁剪、高清倍率、交互态隐藏等细节正确。
5. 对于最多 4 个对象的小工具来说，引入完整编辑器库有些偏重。

所以这个案例更适合用原生 Canvas。我们只保留最小必要抽象：

- 模板区域用 `Point[][]` 描述。
- 图片状态用 `offsetX / offsetY / scale` 描述。
- 裁剪用 `ctx.clip()`。
- 命中检测用多边形点选算法。
- 交互统一走 Pointer Events。
- 导出用离屏 Canvas 重绘。

这样做的好处是实现路径非常可控：每一层数据都和业务强绑定，没有额外的对象系统和编辑器语义。

当然，这并不意味着第三方库不好。下面这些情况就更适合引入 Fabric/Konva：

- 需要支持大量可编辑对象。
- 需要对象旋转、拉伸、组合、锁定、对齐辅助线。
- 需要图层面板、撤销重做、复制粘贴。
- 需要自由绘制路径、文本编辑、贴纸编辑。
- 需要复杂命中检测和选择框交互。
- 团队希望基于成熟画布编辑器能力长期扩展。

换句话说，如果目标是“图片合成工具”，原生 Canvas 足够轻；如果目标是“通用编辑器”，第三方库会更划算。

## 一、用配置描述模板，而不是把布局写死

多人头像合成的核心不是图片本身，而是“每张图应该被裁剪到哪块区域”。如果把布局逻辑写死在绘制函数里，后续增加模板会非常痛苦。

更合适的方式是把模板抽象成一组多边形：

```ts
export interface Point {
    x: number;
    y: number;
}

export interface TemplateConfig {
    img: string;
    templateId: number;
    pathList: Point[][];
}
```

例如一个 2 人模板：

```ts
{
    img: tpl2_1,
    templateId: 1,
    pathList: [
        [
            { x: 0, y: 0 },
            { x: 90, y: 0 },
            { x: 60, y: 150 },
            { x: 0, y: 150 },
        ],
        [
            { x: 92, y: 0 },
            { x: 150, y: 0 },
            { x: 150, y: 150 },
            { x: 62, y: 150 },
        ],
    ],
}
```

这里使用 `150 x 150` 作为逻辑坐标系。无论页面上 canvas 显示多大，绘制逻辑都只关心这个固定坐标系。

这么做有几个好处：

- 模板和渲染逻辑解耦。
- 新增模板只需要增加 `pathList`。
- 不规则斜切、宫格、长条都可以用多边形表达。
- 点击命中检测也可以复用同一份多边形数据。

选择人数和模板的关系则用一个 map 管理：

```ts
export const multiAvatarTemplateMap: Record<number, TemplateConfig[]> = {
    2: [...],
    3: [...],
    4: [...],
};
```

页面层只需要根据当前选择人数取模板：

```ts
const availableTemplates = computed(
    () => multiAvatarTemplateMap[selectedAvatarIds.value.length] ?? [],
);
```

## 二、页面只管状态，Canvas 逻辑抽成 Hook

一开始这类页面很容易写成一个巨大的 `.vue` 文件：头像选择、模板切换、图片加载、Canvas 绘制、拖拽事件、缩放事件、导出逻辑全塞在一起。

可维护性会迅速下降。

因此更好的拆分是：

- `index.vue`：负责业务状态和 UI。
- `useMultiAvatarCanvas.ts`：负责 Canvas 的所有细节。

页面层维护几个核心状态：

```ts
const canvasRef = ref<HTMLCanvasElement | null>(null);
const selectedAvatarIds = ref<number[]>([]);
const selectedTemplateId = ref<number | null>(null);
```

然后把这些状态交给 Hook：

```ts
const {
    canCompose,
    downloadCanvas,
    renderCanvas,
    resetCanvasState,
    resetImageTransforms,
} = useMultiAvatarCanvas({
    canvasRef,
    selectedAvatarIds,
    selectedAvatars,
    selectedTemplate,
});
```

这样页面读起来会非常清楚：

- 选择头像时更新 `selectedAvatarIds`。
- 切换模板时更新 `selectedTemplateId`。
- 监听两者变化后调用 `renderCanvas()`。
- 点击导出时调用 `downloadCanvas()`。

```ts
watch(
    [selectedAvatarIds, selectedTemplateId],
    () => {
        renderCanvas();
    },
    { deep: true },
);
```

这个拆分的价值很直接：Canvas 逻辑可以单独演进，不会污染页面 UI。

## 三、Canvas 高清适配：逻辑尺寸和实际像素分离

Canvas 最大的坑之一是高清屏模糊。

页面中 canvas 的逻辑尺寸是 `150 x 150`，但在 DPR 为 2 的屏幕上，如果实际像素仍然是 `150 x 150`，图像会发糊。

因此初始化 canvas 时需要乘上 `devicePixelRatio`：

```ts
const setupCanvasContext = () => {
    const canvas = canvasRef.value;
    if (!canvas) return null;

    const dpr = window.devicePixelRatio || 1;
    canvas.width = CANVAS_SIZE * dpr;
    canvas.height = CANVAS_SIZE * dpr;

    const ctx = canvas.getContext("2d");
    if (!ctx) return null;
    ctx.setTransform(dpr, 0, 0, dpr, 0, 0);
    return ctx;
};
```

这里有两个尺寸概念：

- `CANVAS_SIZE = 150`：绘制使用的逻辑坐标。
- `canvas.width = 150 * dpr`：真实像素尺寸。

后续所有绘制仍然使用 `0 ~ 150` 的逻辑坐标，不需要到处乘 DPR。

## 四、绘制不规则裁剪区域

头像合成的关键是：每张图片只显示在自己的多边形区域内。

Canvas 里可以通过 `clip()` 实现：

```ts
const drawPolygonPath = (ctx: CanvasRenderingContext2D, points: Point[]) => {
    ctx.beginPath();
    points.forEach((point, index) => {
        if (index === 0) {
            ctx.moveTo(point.x, point.y);
        } else {
            ctx.lineTo(point.x, point.y);
        }
    });
    ctx.closePath();
};
```

绘制单个头像时：

```ts
ctx.save();
drawPolygonPath(ctx, item.polygon);
ctx.clip();
ctx.drawImage(item.image, rect.x, rect.y, rect.width, rect.height);
ctx.restore();
```

注意一定要配合 `save()` / `restore()`。`clip()` 会影响后续绘制，如果不恢复状态，后面的图片也会被裁剪到同一个区域里。

## 五、图片如何填满裁剪区域

每个裁剪区域形状不同，有的是矩形，有的是斜切多边形。为了让头像完整覆盖区域，先计算多边形的外接矩形：

```ts
const getBoundingBox = (points: Point[]) => {
    const xs = points.map((point) => point.x);
    const ys = points.map((point) => point.y);

    const left = Math.min(...xs);
    const top = Math.min(...ys);
    const right = Math.max(...xs);
    const bottom = Math.max(...ys);

    return {
        left,
        top,
        width: right - left,
        height: bottom - top,
    };
};
```

然后使用类似 `object-fit: cover` 的思路：

```ts
const baseScale = Math.max(
    box.width / image.width,
    box.height / image.height,
);
```

`Math.max` 保证图片缩放后一定能覆盖整个区域，不会露出背景。

用户拖拽和缩放后的绘制矩形这样计算：

```ts
const imageScale = baseScale * transform.scale;
const drawWidth = image.width * imageScale;
const drawHeight = image.height * imageScale;

const x = box.left + (box.width - drawWidth) / 2 + transform.offsetX;
const y = box.top + (box.height - drawHeight) / 2 + transform.offsetY;
```

这里的 `transform` 是每个头像独立维护的状态：

```ts
type ImageTransform = {
    offsetX: number;
    offsetY: number;
    scale: number;
};
```

## 六、拖拽时不能把图片拖出裁剪区域

如果允许无限拖动，图片很容易被拖到一边，导致裁剪区域露出黑底。

解决方式是根据图片当前缩放后的尺寸，计算最大可偏移范围：

```ts
const maxOffsetX = Math.max(0, (drawWidth - box.width) / 2);
const maxOffsetY = Math.max(0, (drawHeight - box.height) / 2);

transform.offsetX = clamp(transform.offsetX, -maxOffsetX, maxOffsetX);
transform.offsetY = clamp(transform.offsetY, -maxOffsetY, maxOffsetY);
```

`clamp` 很简单：

```ts
const clamp = (value: number, min: number, max: number) => {
    return Math.min(Math.max(value, min), max);
};
```

这样用户仍然可以自由调整头像位置，但不会破坏“图片覆盖区域”的基本约束。

## 七、Pointer Events 同时支持 PC 和移动端

传统写法可能会分别处理：

- `mousedown`
- `mousemove`
- `mouseup`
- `touchstart`
- `touchmove`
- `touchend`

但现在更推荐使用 Pointer Events。它能统一鼠标、触控笔、触摸屏输入：

```ts
canvas.addEventListener("pointerdown", handleCanvasPointerDown);
canvas.addEventListener("pointermove", handleCanvasPointerMove);
canvas.addEventListener("pointerup", finishCanvasDrag);
canvas.addEventListener("pointercancel", finishCanvasDrag);
```

同时 CSS 中需要禁用浏览器默认触摸行为：

```scss
.compose-canvas {
    touch-action: none;
    user-select: none;
}
```

否则移动端双指缩放或单指拖动时，浏览器可能会滚动页面或触发默认缩放。

## 八、单指拖动图片位置

按下时记录当前指针位置和头像已有偏移：

```ts
dragState.startX = point.x;
dragState.startY = point.y;
dragState.startOffsetX = item.transform.offsetX;
dragState.startOffsetY = item.transform.offsetY;
dragState.isDragging = true;
```

移动时计算增量：

```ts
activeItem.transform.offsetX =
    dragState.startOffsetX + point.x - dragState.startX;
activeItem.transform.offsetY =
    dragState.startOffsetY + point.y - dragState.startY;
```

再调用 `renderCanvas()` 重新绘制即可。

由于最终偏移会在 `getImageDrawRect` 中被 clamp，所以拖动过程也不会露底。

## 九、双指捏合缩放

缩放的核心是记录两根手指的距离变化。

维护一个 `activePointers`：

```ts
const activePointers = new Map<number, Point>();
```

计算前两个指针之间的距离：

```ts
const getPointerDistance = () => {
    const points = Array.from(activePointers.values());
    if (points.length < 2) return 0;

    return Math.hypot(points[0].x - points[1].x, points[0].y - points[1].y);
};
```

双指开始时记录：

```ts
pinchState.startDistance = getPointerDistance();
pinchState.startScale = item.transform.scale;
```

移动时按比例缩放：

```ts
updateImageScale(
    pinchState.activeIndex,
    (pinchState.startScale * distance) / pinchState.startDistance,
);
```

缩放范围也需要限制：

```ts
const MIN_IMAGE_SCALE = 1;
const MAX_IMAGE_SCALE = 4;
```

这样可以避免图片被缩得太小或放得过大。

## 十、PC 端滚轮缩放

PC 上用滚轮缩放会更自然。

实现逻辑是：根据鼠标当前位置找到所在槽位，然后调整该槽位头像的 `scale`：

```ts
const handleCanvasWheel = (event: WheelEvent) => {
    event.preventDefault();
    const point = getCanvasPointer(event);
    if (!point) return;

    const index = getSlotIndexAtPoint(point);
    if (index < 0) return;

    const item = canvasItems.value[index];
    if (!item) return;

    const scaleStep = event.deltaY > 0 ? 0.92 : 1.08;
    updateImageScale(index, item.transform.scale * scaleStep);
};
```

这里有个细节：`wheel` 监听需要设置 `{ passive: false }`，否则 `preventDefault()` 可能不会生效。

```ts
canvas.addEventListener("wheel", handleCanvasWheel, { passive: false });
```

## 十一、拖拽交换头像顺序

除了调整图片位置，这个案例还支持把头像拖到另一个槽位上，完成顺序交换。

关键是命中检测。

因为槽位是不规则多边形，不能简单用矩形判断，需要使用“点是否在多边形内”的算法：

```ts
const isPointInPolygon = (point: Point, polygon: Point[]) => {
    let inside = false;

    for (let i = 0, j = polygon.length - 1; i < polygon.length; j = i++) {
        const xi = polygon[i].x;
        const yi = polygon[i].y;
        const xj = polygon[j].x;
        const yj = polygon[j].y;

        const intersect =
            yi > point.y !== yj > point.y &&
            point.x < ((xj - xi) * (point.y - yi)) / (yj - yi) + xi;

        if (intersect) inside = !inside;
    }

    return inside;
};
```

拖动过程中更新 `hoverIndex`：

```ts
dragState.hoverIndex = getSlotIndexAtPoint(point);
```

松手时，如果拖动距离超过阈值，并且目标槽位不同，就交换两个头像 id：

```ts
const next = [...selectedAvatarIds.value];
[next[fromIndex], next[toIndex]] = [next[toIndex], next[fromIndex]];
selectedAvatarIds.value = next;
```

由于绘制顺序是根据 `selectedAvatarIds` 映射出来的，交换 id 后重新渲染即可。

## 十二、交换状态的视觉反馈

用户拖动时需要知道“当前拖的是谁”和“松手后会交换到哪里”。

绘制时可以给不同状态加覆盖层：

```ts
const isActive = withInteractionState && dragState.activeIndex === index;
const isSwapTarget =
    withInteractionState &&
    dragState.isDragging &&
    dragState.hoverIndex === index &&
    dragState.activeIndex !== index;
```

目标区域加一层半透明色：

```ts
if (isSwapTarget) {
    ctx.save();
    drawPolygonPath(ctx, item.polygon);
    ctx.fillStyle = "rgba(193, 95, 60, 0.24)";
    ctx.fill();
    ctx.restore();
}
```

当前拖动区域则用暗色遮罩和虚线边框：

```ts
if (isActive && dragState.isDragging) {
    ctx.save();
    drawPolygonPath(ctx, item.polygon);
    ctx.fillStyle = "rgba(0, 0, 0, 0.12)";
    ctx.fill();
    ctx.restore();
}
```

这类交互反馈非常重要。否则用户会觉得自己只是在“移动图片”，不确定是否触发了交换。

## 十三、异步图片加载的竞态问题

Canvas 渲染依赖图片加载，而图片加载是异步的。

如果用户快速切换头像或模板，可能出现这种情况：

1. 第一次渲染开始加载图片。
2. 用户马上切换模板，第二次渲染开始。
3. 第一次图片稍后加载完成，反而覆盖了第二次结果。

解决方式是维护一个渲染版本号：

```ts
const renderVersion = ref(0);
```

每次渲染递增：

```ts
const currentVersion = ++renderVersion.value;
```

图片加载完成后检查版本：

```ts
if (currentVersion !== renderVersion.value) return;
```

这是一个很小的保护，但能避免很多“偶现”的渲染错乱。

## 十四、高清导出：不要直接导出预览 Canvas

预览 Canvas 的真实像素尺寸通常跟 DPR 有关。

例如逻辑尺寸是 `150 x 150`，在 DPR 为 2 的屏幕上，预览 Canvas 实际是 `300 x 300`。如果直接：

```ts
canvas.toDataURL("image/png");
```

导出尺寸就会受设备影响，不稳定，也不一定足够清晰。

更好的做法是导出时创建一个离屏 Canvas，固定按 4 倍尺寸重绘：

```ts
const EXPORT_SCALE = 4;

const exportCanvas = document.createElement("canvas");
exportCanvas.width = CANVAS_SIZE * EXPORT_SCALE;
exportCanvas.height = CANVAS_SIZE * EXPORT_SCALE;

const ctx = exportCanvas.getContext("2d");
ctx.setTransform(EXPORT_SCALE, 0, 0, EXPORT_SCALE, 0, 0);
```

然后复用绘制逻辑，但关闭交互态：

```ts
canvasItems.value.forEach((item, index) => {
    drawCanvasItem(ctx, item, index, false);
});
```

最后导出离屏 Canvas：

```ts
const link = document.createElement("a");
link.href = exportCanvas.toDataURL("image/png");
link.download = "multi-avatar-mix.png";
link.click();
```

这样有几个优点：

- 导出尺寸稳定，比如固定 `600 x 600`。
- 不受当前设备 DPR 影响。
- 不会把 hover、拖拽高亮、交换提示画进最终图片。
- 预览 Canvas 可以保持轻量，不需要为了导出而牺牲交互性能。

## 十五、几个值得注意的细节

### 1. Canvas 显示尺寸和绘制尺寸不要混在一起

CSS 控制页面上看起来多大，`canvas.width / canvas.height` 控制真实像素。两者要分清楚。

### 2. 逻辑坐标系要稳定

本案例统一使用 `150 x 150` 的模板坐标。模板配置、命中检测、绘制、拖拽都基于这个坐标系，复杂度会低很多。

### 3. 不规则区域优先用多边形描述

多边形既可以用于 `clip()` 裁剪，也可以用于点击命中检测，一份数据两种用途。

### 4. 手势状态要在卸载时清理

事件监听在 `onMounted` 注册，就要在 `onBeforeUnmount` 移除：

```ts
onBeforeUnmount(() => {
    const canvas = canvasRef.value;
    if (canvas) {
        canvas.removeEventListener("pointerdown", handleCanvasPointerDown);
        canvas.removeEventListener("pointermove", handleCanvasPointerMove);
        canvas.removeEventListener("pointerup", finishCanvasDrag);
        canvas.removeEventListener("pointercancel", finishCanvasDrag);
        canvas.removeEventListener("wheel", handleCanvasWheel);
    }
});
```

### 5. 移动端一定要设置 `touch-action: none`

否则手指拖动 Canvas 时，浏览器可能会滚动页面，导致手势体验失控。

## 总结

这个多人头像合成案例的核心，不是简单地把几张图片画到 Canvas 上，而是把多个问题组合起来解决：

- 用配置驱动模板。
- 用多边形裁剪实现不规则拼图。
- 用 `object-fit: cover` 思路计算图片绘制区域。
- 用独立 transform 保存每张头像的平移和缩放。
- 用 Pointer Events 统一 PC 和移动端输入。
- 用多边形命中检测支持拖拽交换。
- 用离屏 Canvas 固定高清导出尺寸。

如果后续要继续扩展，可以考虑：

- 支持用户上传本地头像。
- 模板配置可视化编辑。
- 增加旋转、镜像、滤镜等图片编辑能力。
- 支持导出不同尺寸和格式。
- 将模板数据从代码迁移到 CMS 或 JSON 配置文件。

Canvas 这类需求的关键在于：先稳定坐标系，再抽象数据结构，最后处理交互细节。只要这三层拆清楚，复杂的图片合成工具也能写得比较干净。
